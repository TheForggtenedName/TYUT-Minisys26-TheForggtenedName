## 一、需求分析

- 哈佛结构存储（指令存储器 + 数据存储器分离）
- 具备简单的 LED 和拨码开关 IO 功能
- 通过行为仿真、综合、实现、比特流生成全流程

**实验二（必做）**：基于 Minisys-1 CPU 的简单接口设计

- 控制器扩展（新增 IO 地址译码能力）
- 数码管外设添加（两个 4 位七段数码管的动态扫描驱动）
- LED 器件完善

### 2.3 IO 地址空间映射（Memory-Mapped I/O）

CPU 通过 LW/SW 指令以统一编址方式访问所有外设，无需专有 IO 指令：

| 地址范围 | 高22位 | 外设 | 片选信号 | 位宽 | 读写 |
|---------|--------|------|---------|------|------|
| `0x00000000 ~ 0x0000FFFF` | `0x000000` | RAM（64KB） | — | 32bit | R/W |
| `0xFFFFFC60 ~ 0xFFFFFC6F` | `0x3FFFFF` | LED | LEDCtrl | 24bit | W |
| `0xFFFFFC70 ~ 0xFFFFFC7F` | `0x3FFFFF` | 拨码开关 | SwitchCtrl | 24bit | R |
| `0xFFFFFC80 ~ 0xFFFFFC8F` | `0x3FFFFF` | 数码管 | DigitubeCtrl | 16bit | W/R |

**地址译码原理**：当 ALU 结果地址的高 22 位全为 1 时（`alu_result[31:10] == 22'h3FFFFF`），判定为 IO 空间访问。memorio 模块进一步根据 bit[7:4] 区分具体外设（6=LED, 7=Switch, 8=Digitube）。

### 2.4 数据通路：一条指令的完整旅程

以 `ORI $1, $0, 0x1D`（机器码 `0x3401001D`）为例：

```
1. ifetc32:   PC=0x00 → ROM[0x00] → instruction=0x3401001D
2. idecode32: 拆出 rs=0, rt=1, imm=0x001D
             读 $0=0, 符号扩展 → sign_extend=0x0000001D
3. control32: opcode=001101 → RegWrite=1, ALUSrc=1, RegDst=0 ...
4. executs32: ALU 算 OR(0, 0x001D) → alu_result=0x0000001D
5. memorio:   地址=0x0000001D, 高22位=0 → RAM 访问 → 不触发 IO
6. idecode32: RegWrite=1 → $1 = ALU结果
```

## 三、详细设计

### 3.1 取指单元（ifetc32.v）

**功能**：根据 PC 从 ROM 取指令；计算下一条指令地址。

**PC+4 计算**：32 位指令 4 字节对齐，PC+4 = 高 30 位加 1，低 2 位恒为 0。
```verilog
assign PC_plus_4[31:2] = PC[31:2] + 1'b1;
assign PC_plus_4[1:0]  = 2'b00;
```

**下一 PC 选择（优先级从高到低）**：
- JR（Jrn=1）：`next_PC = Read_data_1`——跳转到寄存器值
- BEQ（Branch=1 且 Zero=1）：`next_PC = Add_result`——相等则跳转
- BNE（nBranch=1 且 Zero=0）：`next_PC = Add_result`——不等则跳转
- J / JAL：`next_PC = {PC_plus_4[31:28], Instruction[25:0], 2'b00}`——拼接跳转
- 默认：`next_PC = PC_plus_4`——顺序执行

**PC 寄存器**：在时钟下降沿更新。下降沿与 idecode32 的上升沿写寄存器错开半个周期，避免同周期读写冲突。

### 3.2 译码单元（idecode32.v）

**功能**：解析指令字段、读寄存器堆、立即数符号扩展、写回数据选择。

**指令字段提取**：

| 字段 | 位 | 含义 |
|------|-----|------|
| opcode | instruction[31:26] | 操作码 |
| rs | instruction[25:21] | 源寄存器1 |
| rt | instruction[20:16] | 源寄存器2 / I型目标寄存器 |
| rd | instruction[15:11] | R型目标寄存器 |
| immediate | instruction[15:0] | 16位立即数 |

**寄存器堆**：32×32bit，`$0` 恒为 0（写使能时 `write_register_address != 5'b00000` 保护）。

**立即数符号扩展**：`Sign_extend = {{16{Instruction[15]}}, Instruction[15:0]}`。

**写回数据选择（三路 MUX）**：
1. JAL → `opcplus4`（返回地址，写入 $31/$ra）
2. MemtoReg=1 → `read_data`（来自 RAM 或 IO 的读数据）
3. 默认 → `alu_result`（ALU 计算结果）

**实验二 Bug #7 修复**：JAL 指令固定写入 `$31`（而非 rd/rt 字段）——
```verilog
if (Jal) write_register_address = 5'b11111;   // 显式写 $31 ($ra)
else if (RegDst) write_register_address = rd;  // R型选 rd
else write_register_address = rt;              // I型选 rt
```

### 3.3 控制器（control32.v）

**功能**：根据 opcode 和 function 码生成全部控制信号。

**实验一（13 个控制信号）**：

| 信号 | 为1的条件 | 作用 |
|------|----------|------|
| RegDST | R_format (opcode=000000) | 目标寄存器选 rd 而非 rt |
| ALUSrc | I型/LW/SW | ALU 第二操作数选立即数 |
| MemtoReg | LW | 写回数据来自内存 |
| RegWrite | R型或I型或LW或JAL | 写寄存器使能 |
| MemWrite | SW | 内存写使能 |
| Branch | BEQ | 相等分支 |
| nBranch | BNE | 不等分支 |
| Jmp | J | 无条件跳转 |
| Jal | JAL | 跳转并链接 |
| Jrn | R型且Function=001000 | JR寄存器跳转 |
| I_format | opcode[5:3]=001 | 纯 I 型 ALU 指令 |
| Sftmd | R型且Function[5:3]=000 | 移位指令 |
| ALUOp[1:0] | bit1=R或I型, bit0=分支类 | ALU 控制编码 |

**实验二扩展（17 个控制信号）**：

新增 IO 地址译码输入和 4 个访存拆分信号：

```verilog
wire is_io = (Alu_resultHigh[21:0] == 22'h3FFFFF);

MemRead  = Lw && !is_io;    // LW 且非 IO → 读 RAM
IORead   = Lw &&  is_io;    // LW 且是 IO → 读 IO
MemWrite = Sw && !is_io;    // SW 且非 IO → 写 RAM
IOWrite  = Sw &&  is_io;    // SW 且是 IO → 写 IO
MemorIOtoReg = Lw;          // 统一的外部数据写回信号
```

**实验二 Bug #8 修复**：
```verilog
RegWrite = ((R_format && !Jrn) || I_format || Lw || Jal) ? 1 : 0;
```
JR 指令不再错误触发寄存器写。

### 3.4 执行单元（executs32.v）

**功能**：执行 ALU 运算（算术、逻辑、比较、移位）。

**ALU 控制编码（3位）**：

| ALU_ctl | 运算 | 对应指令 |
|---------|------|----------|
| 000 | AND | and, andi |
| 001 | OR | or, ori |
| 010 | ADD | add, addi, lw, sw |
| 011 | ADDU | addu, addiu |
| 100 | XOR | xor, xori |
| 101 | NOR | nor |
| 110 | SUB | sub, beq, bne |
| 111 | SLT/SUBU | slt, slti, subu |

**移位操作**：SLL/SRL/SRA 按 shamt 字段（instruction[10:6]）移位；SLLV/SRLV/SRAV 按 rs 寄存器值移位。

**分支地址**：`Add_Result = PC_plus_4 + (Sign_extend << 2)`

**Zero 标志**：`Zero = (ALU_output_mux == 0) ? 1 : 0`

### 3.5 存储器系统

#### 3.5.1 dmemory32.v —— 数据 RAM 封装

封装 Vivado Block Memory Generator IP（Single Port RAM, 32bit×16384 = 64KB）。

**时钟反相**：`wire ram_clk = !ram_clk_i;`——Block RAM 读操作有硬件延迟，反相时钟将读数据"提前"半个周期，确保 CPU 在同一周期锁存到正确数据。

**CPU/UART 双模式切换**：
```verilog
wire kickOff = upg_rst_i | (~upg_rst_i & upg_done_i);
```
- kickOff=0（UART 烧写中）：RAM 使用 UART 时钟和地址
- kickOff=1（烧写完成或复位）：RAM 切回 CPU 时钟和地址

#### 3.5.2 memorio.v —— 存储器/IO 地址路由

**四个核心功能**：

1. **地址透传**：`assign address = caddress;`
2. **写数据路由**：`memwrite=1` 或 `iowrite=1` 时输出 `wdata`，否则输出高阻 `32'hZZZZZZZZ`
3. **读数据路由**：`assign rdata = (memread==1) ? mread_data : {16'h00, ioread_data[15:0]};`
4. **IO 片选译码**：
```verilog
assign LEDCtrl     = ((iorw==1) && (caddress[31:4] == 28'hFFFFFC6)) ? 1 : 0;
assign SwitchCtrl  = ((iorw==1) && (caddress[31:4] == 28'hFFFFFC7)) ? 1 : 0;
assign DigitubeCtrl = ((iorw==1) && (caddress[31:4] == 28'hFFFFFC8)) ? 1 : 0;  // 实验二新增
```

取 `caddress[31:4]` 比较而非 `caddress[31:0]`，是因为低 4 位是外设内部偏移（LED 分两次写入需要地址偏移），不应参与基地址比较。

#### 3.5.3 ioread.v —— IO 读数据汇聚

多路选择器，将各 IO 设备的读数据汇总返回 CPU。实验二新增数码管读通道并修复 latch：
```verilog
always @* begin
    if (reset)          ioread_data = 16'b0;
    else if (ior) begin
        if (switchctrl)     ioread_data = ioread_data_switch;
        else if (digitubectrl) ioread_data = ioread_data_digitube;
        else                ioread_data = 16'b0;  // 修复 latch
    end else             ioread_data = 16'b0;
end
```

### 3.6 外设模块

#### 3.6.1 leds.v —— LED 输出

24 位 LED 输出。基地址 `0xFFFFFC60`，分两次写入（偏移 0 写低 16 位，偏移 4 写高 8 位）。

#### 3.6.2 switchs.v —— 拨码开关输入

24 位开关输入。基地址 `0xFFFFFC70`，分两次读取（偏移 0 读低 16 位，偏移 4 读高 8 位）。下降沿锁存（与 LED 的上升沿错开）。

#### 3.6.3 digitube.v —— 数码管驱动（实验二新建）

**设计原理**：两个 4 位七段数码管共享段选信号线，通过动态扫描交替点亮。13 位计数器最高位（~2.8kHz）控制选通交替，肉眼无闪烁感。

**写入格式**：`write_data[7:4]` → 左数码管值，`write_data[3:0]` → 右数码管值。

**段码译码（共阳极，0=亮）**：

| 值 | 显示 | seg[7:0] |
|----|------|----------|
| 0 | 0 | 11000000 |
| 1 | 1 | 11111001 |
| 2 | 2 | 10100100 |
| 3 | 3 | 10110000 |
| 4 | 4 | 10011001 |
| 5 | 5 | 10010010 |
| 6 | 6 | 10000010 |
| 7 | 7 | 11111000 |
| 8 | 8 | 10000000 |
| 9 | 9 | 10010000 |
| A-F | A-F | 标准七段码 |

### 3.7 顶层集成（minisys.v）

**关键设计点**：

1. **复位逻辑**：`assign rst = fpga_rst | !upg_rst;`——板级复位或 UART 烧写未就绪时 CPU 复位
2. **BUFG 时钟缓冲**：`BUFG U1(.I(start_pg), .O(spg_bufg));`——非时钟信号走 BUFG 需要 `CLOCK_DEDICATED_ROUTE FALSE` 约束
3. **9 个模块实例化**：每个端口名与 wire 名一一对应，模板中约 59 处 `???` 占位符逐一手工补齐
4. **数码管顶层端口（实验二新增）**：`output [7:0] seg`（段选）、`output [3:0] an`（位选）

### 4.2 实验二：队友代码审查

6月4日上午接收队友实验二提交物，逐文件审查发现 **13 个问题**，分为四类：

- **致命缺失（4 个）**：digitube.v 不存在、xdc 无数码管引脚、memorio 无 DigitubeCtrl、ioread 无 digitube 通道
- **集成错误（4 个）**：control32 新端口全被注释（最致命——导致全部 IO 功能失效）、idecode32 端口悬空、is_io 信号定义冗余、仿真文件命名错误
- **逻辑错误（2 个）**：JR RegWrite 逻辑不完备、JAL 写回 Bug 未修复
- **匹配问题（3 个）**：端口命名不一致、顶层端口未扩展

队友完成度评估约 20%，决定不以半成品修补，基于实验一已验证交付包从零推演实验二。

### 4.3 仿真调试关键经验

1. **PLL 锁定时间**：cpuclk IP 的 MMCM 上电后需 5-7μs 锁定，仿真必须 `#7000 rst = 0` 等够时间
2. **`get_value` 局限**：控制信号（branch/jmp）仅在译码周期约 43ns 内为 1，`get_value` 快照几乎无法命中。应使用 `when {signal==1} {stop}` 自动捕获脉冲
3. **PC 轨迹验证**：观察 PC 是否按预期跳转是最高效的仿真验证方式——如果 PC 轨迹正确，取指和译码基本正确
4. **实验二仿真瓶颈**：原测试程序含 `DELAY=0x3F9409≈416万次` 迭代延迟，XSim 不支持条件断点，改用专用数码管测试程序（digitube_test.asm）验证

### 4.4 综合/实现问题

| 阶段 | 实验一 | 实验二 |
|------|--------|--------|
| 综合 | 0 Error | 0 Error |
| 优化 | 0 Error | 0 Error, 0 Warning |
| 布局 | WNS=95.771ns, 0 Error | WNS=0.911ns, 0 Error |
| 布线 | WNS=95.342ns, WHS=0.069, 0 unrouted | WNS=-1.843ns, 0 unrouted |
| 比特流 | 生成成功 | 待补数码管引脚约束 |

WNS=-1.843ns：发生在 UART 编程器（10MHz）和 CPU（23MHz）同时访问 BRAM 的跨时钟域路径。实际下板时两者不会同时工作，不影响功能。

## 五、用户手册

### 5.1 工程创建

1. Vivado 2019.2 → Create Project → 名称 `Minisys1_CPU`
2. 芯片选型：**xc7a100tfgg484-1**（Artix-7, FGG484 封装）
3. 添加 14 个 Verilog 源文件到 Design Sources
4. 添加 `minisys_sim.v` 到 Simulation Sources
5. 添加 `uart_bmpg.edif` 到 Design Sources
6. 添加 `minisys.xdc` 到 Constraints

### 5.2 IP 核配置

| IP 名称 | 类型 | 关键参数 |
|---------|------|---------|
| cpuclk | Clocking Wizard v6.0 | 100MHz→23MHz(CPU) + 10MHz(UART) |
| prgrom | Block Memory Generator v8.4 | Single Port ROM, 32bit×16384, 加载 prgmip32.coe |
| ram | Block Memory Generator v8.4 | Single Port RAM, 32bit×16384, 加载 dmem32.coe |

**注意**：.coe 文件路径必须全英文（Vivado 2019.x 对非 ASCII 路径不兼容）。

### 5.3 仿真运行

1. 生成 IP 仿真目标：`generate_target simulation [get_ips]`
2. Flow Navigator → Run Simulation → Run Behavioral Simulation
3. Tcl Console 中 `run 20000ns`（默认 1000ns 不够 PLL 锁定）
4. 关键观察信号：`clock`, `rst`, `instruction`, `pc_plus_4`, `alu_result`, `iowrite`, `digitubectrl`

### 5.4 综合与实现（命令行方式）

```tcl
synth_design -top minisys -part xc7a100tfgg484-1
opt_design
place_design
route_design
write_bitstream -force <路径>/minisys.bit
```

**注意**：不要使用 `launch_runs`（其缓存的 tcl 脚本可能写死旧芯片型号）。

### 5.5 下板测试

1. 使用 Minisys 汇编器（`MinisysAv2.0.exe`）将 .asm 汇编为 .coe 文件
2. 将 .coe 加载到 prgrom IP，重新生成比特流
3. USB-JTAG 下载到 Minisys 实验板
4. 预期结果：
   - LED 按程序逻辑亮/灭
   - 拨码开关值可通过 LW 指令读入寄存器
   - 数码管交替动态扫描显示，肉眼无闪烁

### 8.1 技术成果

本项目完成了两个实验的全部设计目标：

- **实验一**：实现了一颗 32 位单周期 MIPS CPU，覆盖 31 条指令，通过行为仿真、综合、实现、比特流生成全流程闭环
- **实验二**：在实验一基础上完成控制器 IO 地址译码扩展和数码管外设添加，综合和实现均通过

### 8.2 经验教训

1. **模板代码不可轻信**——官方教辅资料包含重复 endmodule、注释掉的约束、命名不一致等问题
2. **仿真通过 ≠ 代码正确**——Bug #7 和 #8 在仿真中完全不可见，靠人工审查才发现
3. **"恰好不暴露"是最危险的 Bug**——`$0` 写保护同时"救"了两个 Bug，纯属运气
4. **Vivado 工程路径必须全英文**——中文路径导致 .coe 加载失败
5. **团队协作需要强集成**——队友提交物完成度仅 20%，最终基于实验一从零推演实验二
6. **芯片封装必须确认无误**——CSG324≠FGG484，选错导致所有引脚报错
7. **PLL/MMCM 锁定时间影响仿真**——仿真前几微秒 clock=0 是正常现象，需等待
