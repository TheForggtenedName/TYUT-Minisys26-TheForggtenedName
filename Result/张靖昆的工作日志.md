（3）理解控制器扩展中访存信号的拆分逻辑：

实验一中只有 `MemtoReg=Lw` 和 `MemWrite=Sw` 两个访存信号，无法区分内存和IO。实验二引入 `is_io` 条件（`Alu_resultHigh[21:0] == 22'h3FFFFF`）后拆分为：
- `MemRead = Lw && !is_io`——LW指令 + 非IO地址 → 读RAM
- `IORead = Lw && is_io`——LW指令 + IO地址 → 读外设
- `MemWrite = Sw && !is_io`——SW指令 + 非IO地址 → 写RAM
- `IOWrite = Sw && is_io`——SW指令 + IO地址 → 写外设

**设计思路**：CPU统一用LW/SW指令操作内存和外设（Memory-Mapped I/O），硬件通过地址高位判断访问目标。同一条LW指令，访问0x00001000时MemRead=1（读RAM），访问0xFFFFFC80时IORead=1（读数码管）。

（1）**新建 controlIO32.v——扩展控制器（17端口）**

模块声明（与实验一的 control32.v 对比）：

| 对比项 | control32.v（实验一） | controlIO32.v（实验二） |
|--------|---------------------|----------------------|
| 输入 | Opcode, Function_opcode | Opcode, Function_opcode, **Alu_resultHigh[21:0]** |
| 输出数 | 13个 | **17个** |
| 关键变化 | MemtoReg, MemWrite | MemRead, MemWrite, IORead, IOWrite, **MemorIOtoReg** |

核心实现逻辑：

```verilog
// IO地址译码——高22位全1 → IO空间
wire Is_IO_Access;
assign Is_IO_Access = (Alu_resultHigh[21:0] == 22'h3FFFFF) ? 1'b1 : 1'b0;
// 22'h3FFFFF = 22位全1 = 二进制1111111111111111111111
// 对应地址空间 0xFFFFFC00 ~ 0xFFFFFFFF
```

访存信号拆分：
```verilog
assign MemWrite = Sw && !Is_IO_Access;  // SW到RAM地址 → 写内存
assign IOWrite  = Sw && Is_IO_Access;   // SW到IO地址  → 写外设
assign MemRead  = Lw && !Is_IO_Access;  // LW从RAM地址 → 读内存
assign IORead   = Lw && Is_IO_Access;   // LW从IO地址  → 读外设
assign MemorIOtoReg = Lw;               // LW指令都需要写回寄存器
```

Bug #8修复——JR指令排除RegWrite：
```verilog
// 实验一（错误）：
assign RegWrite = (R_format || I_format || Lw || Jal) ? 1'b1 : 1'b0;
// JR属于R型，R_format=1 → RegWrite=1，但JR不写寄存器！

// 实验二（修复）：
assign RegWrite = ((R_format || I_format || Lw || Jal) && !Jrn) ? 1'b1 : 1'b0;
// JR时Jrn=1 → RegWrite=0，不再错误触发写寄存器
```

（2）**新建 digitube.v——数码管驱动模块**

设计参数：
- 输入时钟：23MHz（CPU时钟）
- 扫描方式：计数器高位选通，23MHz ÷ 2^20 ≈ 22Hz——但20位计数器只用高2位（[19:18]），实际扫描频率约23MHz ÷ 2^18 ≈ 88Hz，或直接用更快的计数器位
- 段码表：16个值（0-9, A-F），共阳极编码（后续可能调整为共阴极）
- 写入格式：支持多种写入模式——偏移0写digit0，偏移1写digit1，偏移2同时写两个

模块接口：
| 端口 | 方向 | 位宽 | 说明 |
|------|------|------|------|
| tube_clk | input | 1 | 系统时钟 23MHz |
| tube_rst | input | 1 | 复位（高有效） |
| tube_write | input | 1 | 写使能（来自IOWrite） |
| tube_cs | input | 1 | 片选（来自DigitubeCtrl） |
| tube_addr | input | [2:0] | 地址偏移 |
| tube_wdata | input | [15:0] | 写入数据 |
| tube_sel | output | [7:0] | 段选信号（a-g + dp） |
| tube_digit_sel | output | [1:0] | 位选信号（2个数码管） |

关键设计点——动态扫描：两个数码管共享段选信号线（tube_sel[7:0]），硬件限制不能同时亮不同内容。通过计数器高位交替选通：scan_sel=00→digit0选通，scan_sel=01→digit1选通。扫描频率约为23MHz÷2^18≈88Hz，远超50Hz闪烁阈值。

段码表（共阳极/active high版本）：
```
0: 00111111 (abcdef)     8: 01111111 (abcdefg)
1: 00000110 (bc)         9: 01101111 (abcdfg)
2: 01011011 (abdeg)      A: 01110111 (abcefg)
3: 01001111 (abcdg)      B: 01111100 (cdefg)
4: 01100110 (bcfg)       C: 00111001 (adef)
5: 01101101 (acdfg)      D: 01011110 (bcdeg)
6: 01111101 (acdefg)     E: 01111001 (adefg)
7: 00000111 (abc)        F: 01110001 (aefg)
```

（3）**修改 memorio.v——新增数码管片选**

在原有 LEDCtrl 和 SwitchCtrl 的基础上，增加第三路片选：
```verilog
// Digital tube address range: 0xFFFFFC80-0xFFFFFC8F
assign DigitubeCtrl = ((iorw==1) && (caddress[31:4] == 28'hFFFFFC8)) ? 1'b1:1'b0;
```

三路片选的地址范围：
- LEDCtrl：caddress[31:4] == 28'hFFFFFC6 → 0xFFFFFC60~0xFFFFFC6F
- SwitchCtrl：caddress[31:4] == 28'hFFFFFC7 → 0xFFFFFC70~0xFFFFFC7F
- DigitubeCtrl：caddress[31:4] == 28'hFFFFFC8 → 0xFFFFFC80~0xFFFFFC8F

取 [31:4] 比较而非 [31:0] 的原因：低4位是外设内部偏移量（LED分两次写需要偏移量），不应参与基地址比较。

（4）**修改 ioread.v——新增数码管读通道 + 修复latch**

实验一的 ioread.v 有两个问题：①只有switch读通道；②组合逻辑自赋值产生latch。

实验二修复后的完整代码：
```verilog
always @* begin
    if(reset == 1)
        ioread_data = 16'b0;
    else if(ior == 1) begin
        if(switchctrl == 1)
            ioread_data = ioread_data_switch;
        else if(digituberead == 1)       // 新增：数码管读通道
            ioread_data = ioread_data_digitube;
        else
            ioread_data = 16'h0000;      // 修复latch：确定值而非自赋值
    end
    else
        ioread_data = 16'h0000;          // 修复latch
end
```

### 3. 存在的问题以及解决办法
（1）**digitube.v 的段码编码方向**：GitHub版本（active high, 1=亮）与本地交付包版本（active low / 共阳极, 0=亮）编码方向相反。需在实际下板后确认数码管理号。两种编码只是取反关系，修改成本很低。

（2）**controlIO32.v 的命名选择**：设计中选择新建 controlIO32.v 而非修改实验一的 control32.v。理由是——实验一是独立的CPU设计，实验二是增量扩展；保留 control32.v 作为基础版本，controlIO32.v 作为扩展版本，文件命名清晰区分了两次实验的差异。

（3）**ioread.v 的 latch 修复**：这是一个从实验一继承的低优先级问题（原模板自赋值写法在组合逻辑中推断锁存器）。实验二中所有else分支赋 `16'h0000`，彻底消除latch。

### 2. 工作内容
（1）**控制器扩展章节撰写**

核心设计原则——**从"一刀切"到"精细分类"**：

实验一的控制器只有内存概念——`MemtoReg=Lw` 和 `MemWrite=Sw` 两个信号足够。实验二引入了IO设备，同一条LW/SW指令可能访问内存也可能访问外设——控制器必须"知道"当前地址属于哪个空间，并输出不同的控制信号。

IO地址译码：
```
Alu_resultHigh[21:0] == 22'h3FFFFF → Is_IO_Access = 1
所有IO外设地址都在 0xFFFFFC00 以上 → 高22位全1
22'h3FFFFF = 22'b1111111111111111111111
```

访存信号拆分的设计逻辑：
```
原来：MemtoReg=Lw, MemWrite=Sw
现在：MemRead=Lw && !IO, IORead=Lw && IO
      MemWrite=Sw && !IO, IOWrite=Sw && IO
```

四个新信号两两互斥（MemRead和IORead同一时刻最多一个为1），确保数据总线不冲突。

（2）**IO地址空间完整映射章节撰写**

Memory-Mapped I/O的设计原理：
- CPU没有专门的IN/OUT指令——直接用LW/SW访问特定地址操作外设
- 优点：指令集简洁，编译器无需特殊处理IO
- 硬件实现：地址译码器（memorio模块）根据高位地址产生片选信号

IO地址空间的位域划分：
```
[31:10] — 高22位 → 区分RAM(全0)和IO空间(全1)
[9:4]   — 中间6位 → 区分不同外设（6=LED, 7=Switch, 8=Digitube）
[3:0]   — 低4位   → 外设内部偏移（LED分两次写：偏移0低16位,偏移4高8位）
```

这个位域划分解释了为什么memorio中用 `caddress[31:4]` 比较基地址——低4位不属于基地址，属于外设内部寻址。

（3）**数码管章节撰写**

七段数码管的基本原理：
```
    a
   ---
f |   | b
   -g-
e |   | c
   ---
    d    dp(小数点)
```

动态扫描的必要性：
- 两个数码管的a-g段选线是共享的（硬件连线并联）
- 如果同时给两个数码管不同的段码 → 两个管都显示乱七八糟
- 解决方案：交替点亮——时刻1点亮digit0给段码0，时刻2点亮digit1给段码1
- 频率>50Hz → 人眼看到的是"两个管同时稳定亮"

段码编码：
- 每个数码管7段（a-g）+ 小数点（dp）= 8位段选信号
- 通过查表将4位BCD值转换为8位段码
- 当前按active high编码：1=段亮, 0=段灭

（4）**IO路由扩展章节撰写**

memorio.v在实验二中的角色升级：
- 实验一：RAM/LED/Switch 三路路由
- 实验二：RAM/LED/Switch/数码管 四路路由

ioread.v在实验二中的角色升级：
- 实验一：仅Switch读通道（+自赋值latch）
- 实验二：Switch + 数码管双通道 + latch修复

minisys.v中的信号流：
```
controlIO32 → IOWrite → memorio.iowrite → digitube.tube_write
controlIO32 → IORead  → ioread.ior
alu_result[31:10] → Alu_resultHigh → controlIO32.Is_IO_Access
memorio.DigitubeCtrl → digitube.tube_cs + ioread.digituberead
```
建立了完整的"控制器→地址译码→外设选通"信号链。

（2）**答辩准备——角色D核心问题**

**核心问题1：为什么实验二需要把MemWrite拆成MemWrite和IOWrite？**

实验一中CPU只有一种外部设备——内存。SW指令就是"写内存"。实验二增加了LED和数码管——同一条SW指令，写0x00001000是写内存，写0xFFFFFC80是写数码管。硬件必须区分这两种情况——MemWrite控制RAM的写使能，IOWrite控制外设的写使能，两者互斥（同一时刻最多一个为1）。

如果不用拆分信号，memorio模块需要自己判断地址来决定数据路由——这会让地址译码逻辑分散在多个模块中。集中到controlIO32中判断（通过Alu_resultHigh → Is_IO_Access），所有下游模块只需根据信号名就能知道该做什么。

**核心问题2：数码管为什么不能两个同时亮？**

因为硬件连线——两个数码管的a-g段选线是并联的（共享8根段选线）。如果同时给两个数码管不同段码，8根线上只能有一个电压值，两个管看到的都是同一个段码。解决方案是动态扫描——以>50Hz的频率交替点亮，人眼因视觉暂留效应看到的是"两个管同时亮"。

**核心问题3：Is_IO_Access的判断条件为什么是22'h3FFFFF？**

22位全1的二进制是 `1111111111111111111111`。放在32位地址的[31:10]位置，等于地址的高22位全1。这22位全1对应的最小地址是 `0xFFFFFC00`（低10位全0）。所有IO外设都在这个地址以上——LED 0xFFFFFC60、Switch 0xFFFFFC70、Digitube 0xFFFFFC80。RAM地址的高22位是0x000000（64KB范围内）。所以`高22位==全1`完美区分了IO空间和RAM空间。

**核心问题4：实验二给实验一新增了什么？**

- 新增1个控制器（controlIO32.v, 66行）——IO地址译码 + 访存信号拆分

- 修复3个Bug（Bug #7 JAL写回、Bug #8 JR RegWrite、ioread latch）

（3）**个人技术收获总结**

1. **Memory-Mapped I/O的完整实现**：从理论（统一地址空间）到硬件（controller判断+memorio译码+外设响应），完整经历了一个硬件IO子系统的设计过程。核心认识：CPU视角"统一地址空间"的简洁性是靠硬件层的"地址译码器"实现的——软件简单，硬件负责区分。

2. **控制器的设计演进**：从实验一的13端口到实验二的17端口，不是简单增加信号，而是"信号语义的精细化"——原来的MemtoReg是一个笼统的"外部数据来了"，拆成MemRead/IORead/MemorIOtoReg后每个信号含义单一明确。好的控制器输出信号就像好的API——一个信号只做一件事。

3. **数码管的"硬件约束"与"软件补偿"**：两个数码管共享段选线的硬件限制，通过动态扫描（软件/硬件协同）来克服。88Hz的扫描频率是硬件计数器实现的，但"交替点亮"的逻辑是人设计的——这是典型的硬件-固件边界问题。

4. **地址译码的位域设计**：`caddress[31:4]` 分两部分——高22位[31:10]区分RAM/IO（controlIO32的Is_IO_Access），中6位[9:4]区分不同外设（memorio的LEDCtrl/SwitchCtrl/DigitubeCtrl），低4位[3:0]在外设内部使用（LED/数码管的分两次访问）。这种位域划分让地址空间既紧凑又可扩展——如果将来需要第4个外设（如VGA），基地址取0xFFFFFC90即可，只需在memorio加一行地址比较。

5. **组合逻辑完备性与latch**：ioread.v的自赋值bug让我深刻理解了FPGA设计的一个基本原则——组合逻辑的always块必须覆盖所有分支（完备性），否则综合工具推断锁存器。`else ioread_data = ioread_data` 在Verilog中就是"保持上一值"，硬件只能用latch实现。正确的写法是在块首或所有else分支赋默认值。

6. **团队协作中的接口定义**：作为角色D，我新建的controlIO32.v需要同时被minisys.v（顶层）、memorio.v（IO路由）、ioread.v（读汇聚）三个模块引用。只要17个端口的名称和位宽定义清楚，三方可以独立开发——模块化设计的前提就是接口先约定好。

### 3. 存在的问题以及解决办法
（1）**数码管段码极性未实测**：GitHub版使用active high（1=亮），是否与Minisys实验板匹配未知。修改成本极低——只需将所有16个段码值按位取反即可。

（2）**数码管引脚约束缺失**：seg[7:0]和an[3:0]共12个FPGA引脚需从Minisys板原理图获取。补充后直接 `write_bitstream` 即可生成最终比特流，无需重新综合实现。
