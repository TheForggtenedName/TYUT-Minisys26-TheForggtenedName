（2）**idecode32.v Bug #7 修复验证**

配合角色 C 验证 JAL 写回目标寄存器修复的正确性：

原 Bug 分析（从实验一继承）：
```
// 错误代码：
if (RegDst == 1'b1)
    write_register_address = rd;     // R 型选 rd
else
    write_register_address = rt;     // I 型选 rt
// JAL 指令 RegDst=0 → 走 else → 选了 instruction[20:16]
// 但 JAL 的 bit[20:16] 是跳转地址的一部分，不是寄存器号！
```

修复后代码：
```verilog
if (Jal == 1'b1)
    write_register_address = 5'b11111;     // JAL 显式写 $31 ($ra)
else if (RegDst == 1'b1)
    write_register_address = rd;           // R 型选 rd
else
    write_register_address = rt;           // I 型选 rt
```

从寄存器文件角度确认修复正确：`$31`（即 `$ra`）是可写寄存器，JAL 保存的返回地址 `opcplus4` 写入 `$31` 后可由 JR 指令（`next_PC = Read_data_1`）正确读出跳转——JAL/JR 的调用-返回机制完全依赖于正确的寄存器写入。

（2）**ALU 控制编码生成逻辑深入分析**

executs32.v 中 ALU 控制码的生成涉及两层映射：

**第一层：control32 根据 opcode 生成 ALUOp[1:0]**
| ALUOp[1:0] | 含义 | 触发条件 |
|------------|------|---------|
| 00 | 不需要 ALU 运算 | — |
| 01 | 分支类（需减法比较） | BEQ 或 BNE |
| 10 | R 型（由 Function 决定） | R_format=1 |
| 11 | I 型立即数运算 | I_format=1 |

**第二层：executs32 根据 ALUOp + Function_opcode 生成 ALU_ctl[2:0]**

当 ALUOp=10（R型）时，根据 Function_opcode[3:0] 生成：
| Function[3:0] | 指令 | ALU_ctl |
|--------------|------|---------|
| 0000 (AND) | and | 000 |
| 0001 (OR) | or | 001 |
| 0010 (ADD) | add | 010 |
| 0011 (ADDU) | addu | 011 |
| 0100 (XOR) | xor | 100 |
| 0101 (NOR) | nor | 101 |
| 0110 (SUB) | sub | 110 |
| 0111 (SLT) | slt | 111 |
| 1000 (JR) | jr | — |

当 ALUOp=11（I型）时，直接由 opcode[2:0] 映射：
| opcode[2:0] | 指令 | ALU_ctl |
|-------------|------|---------|
| 000 | andi | 000 |
| 001 | ori | 001 |
| 010 | addi | 010 |
| 011 | addiu | 011 |
| 100 | xori | 100 |
| 110 | slti | 111 |

当 ALUOp=01（分支）时，ALU_ctl=110（SUB），通过减法结果判断相等/不等。

（3）**移位操作的控制逻辑**

Sftmd 信号（来自 control32）指示当前为移位指令（R_format=1 且 Function[5:3]=000）。Sftmd=1 时 ALU 输出选择移位结果而非 ALU 运算结果。

移位类型由 Function_opcode[2:0] 决定：

| Function[2:0] | 指令 | 操作 |
|--------------|------|------|
| 000 | SLL | `Binput << Shamt`（逻辑左移） |
| 010 | SRL | `Binput >> Shamt`（逻辑右移） |
| 011 | SRA | `$signed(Binput) >>> Shamt`（算术右移） |
| 100 | SLLV | `Binput << Ainput[4:0]`（可变逻辑左移） |
| 110 | SRLV | `Binput >> Ainput[4:0]`（可变逻辑右移） |
| 111 | SRAV | `$signed(Binput) >>> Ainput[4:0]`（可变算术右移） |

关键理解：SLL/SRL/SRA 的移位量来自指令中的 shamt 字段（instruction[10:6]），SLLV/SRLV/SRAV 的移位量来自 rs 寄存器的低 5 位。SRA/SRAV 必须用 `$signed()` 包裹 Binput，编译器才知道这是带符号数，右移时高位填符号位。

### 3. 存在的问题以及解决办法
无新问题。ALU 控制编码的理解已经比较透彻，两层映射（opcode→ALUOp + ALUOp+Function→ALU_ctl）的设计是为了减少控制信号位宽——如果不用两层映射，control32 需要直接输出 3 位 ALU_ctl，每位的译码逻辑会更复杂。

### 3. 存在的问题以及解决办法
（1）**ANDI/ORI/XORI 的零扩展 vs 符号扩展**：当前 executs32.v 中使用 `Sign_extend` 作为立即数操作的 B 输入，对 ANDI/ORI/XORI 这三个逻辑立即数指令，MIPS 规范要求零扩展。当前恰好正确是因为立即数 bit15=0 时两者相同，但 bit15=1 的立即数（如 ORI $1,$0,0xFFFF）在符号扩展后变成 0xFFFFFFFF，再 OR 0 结果不对。实际测试程序中 ANDI/ORI/XORI 使用的立即数都在 0~65535 范围内且 bit15 不一定为 0——**这是一个需要留意的潜在问题**。当前 31 条指令测试程序未使用 bit15=1 的 ANDI/ORI/XORI 立即数，因此未暴露。临时解决方案：在使用 ANDI/ORI/XORI 时确保立即数 bit15=0（即值在 0-32767 范围内）。根本解决方案：在 executs32.v 中对 I_format 的逻辑指令使用零扩展。

**ALU 控制编码（3 位）完整映射**：
| ALU_ctl[2:0] | 运算 | 数学表达 | 对应指令 |
|--------------|------|---------|---------|
| 000 | AND | A & B | and, andi |
| 001 | OR | A \| B | or, ori |
| 010 | ADD | A + B | add, addi, lw, sw |
| 011 | ADDU | A + B | addu, addiu |
| 100 | XOR | A ^ B | xor, xori |
| 101 | NOR | ~(A \| B) | nor |
| 110 | SUB | A - B | sub, beq, bne |
| 111 | SLT/SUBU | A < B / A - B | slt, slti, subu |

**特殊操作**：
- SLT：`($signed(A) < $signed(B)) ? 32'd1 : 32'd0`——有符号比较
- SLTIU：通过 SU BU 实现（ALU_ctl=111），利用 ALU 溢出来判断无符号数大小
- LUI：`{Binput[15:0], 16'b0}`——直接将 16 位立即数放到高 16 位

（2）**移位操作章节撰写**

6 种移位指令分为两组：

**固定移位（移位量来自 shamt 字段）**：
| 指令 | 操作 | 说明 |
|------|------|------|
| SLL | `Binput << Shamt` | 逻辑左移，低位填 0 |
| SRL | `Binput >> Shamt` | 逻辑右移，高位填 0 |
| SRA | `$signed(Binput) >>> Shamt` | 算术右移，高位填符号位 |

**可变移位（移位量来自 rs 寄存器低 5 位）**：
| 指令 | 操作 | 说明 |
|------|------|------|
| SLLV | `Binput << Ainput[4:0]` | 可变逻辑左移 |
| SRLV | `Binput >> Ainput[4:0]` | 可变逻辑右移 |
| SRAV | `$signed(Binput) >>> Ainput[4:0]` | 可变算术右移 |

**关键 Verilog 知识点**：SRA/SRAV 必须用 `$signed(Binput) >>> Shamt`，不能直接用 `Binput >>> Shamt`。因为在 Verilog 中，`>>>` 的行为取决于左操作数是否声明为 `signed`。如果不加 `$signed()`，综合工具可能按照逻辑右移处理（高位填 0），导致算术右移功能错误。

| 特性 | 实现 |
|------|------|
| 寄存器数量 | 32 个 × 32 位 |
| 读端口 | 2 个（组合逻辑，无延迟） |
| 写端口 | 1 个（时钟上升沿） |
| `$0` 处理 | 硬编码为 0，写使能门控排除 `$0` |
| 读操作 | `read_data_1 = register[rs]; read_data_2 = register[rt];` |
| 写操作 | `register[rd/rt] <= write_data`（RegWrite=1 且 addr≠0） |

（4）**Zero 标志与分支地址计算**

- Zero 标志：`Zero = (ALU_output_mux == 0) ? 1'b1 : 1'b0`，供 BEQ/BNE 的条件判断
- 分支地址：`Add_Result = PC_plus_4 + (Sign_extend << 2)`，PC 相对寻址，16 位偏移量左移 2 位后与 PC+4 相加

5. 确认 Zero 标志和 Add_Result 的计算逻辑

**移位操作是最容易出错的地方**——SRA 让我学了 `$signed()` 的用法。如果不加 `$signed()`，Verilog 的 `>>>` 对无符号数的行为与 `>>` 完全一样（高位填 0），算术右移就变成了逻辑右移，功能完全错误。

（2）**数据通路中 ALU 的角色梳理**

以 `ADD $3, $1, $2`（R 型，机器码 `0x00221820`）为例：

```
1. ifetc32:  PC=0x?? → ROM 输出 0x00221820 → instruction
2. idecode32: 拆出 rs=$1, rt=$2, rd=$3
             读 read_data_1=$1 的值, read_data_2=$2 的值
3. control32: R_format=1 → RegDst=1, ALUSrc=0, ALUOp=10
4. executs32: A=read_data_1, B=read_data_2(ALUSrc=0)
             ALUOp=10 + Function=100000(ADD) → ALU_ctl=010
             ALU_output_mux = A + B → alu_result
5. memorio:  地址不触发 IO → 不写入外设
6. idecode32: RegWrite=1 → register[$3] = alu_result（ALU 结果写回）
```

ALU 在数据通路中处于"计算核心"的位置——上游是译码单元提供的操作数，下游是写回寄存器的结果。ALU 本身不关心指令类型，只根据控制码机械执行运算。

（3）**寄存器文件的 $0 保护与实验一潜伏 Bug 的关系分析**

在实训报告中详细分析了 `$0` 写保护如何同时"掩盖"了两个 Bug：
- Bug #7（JAL 写回）：JAL 目标地址 bit[20:16]=0 → 选了 $0 → 写保护拦截
- Bug #8（JR RegWrite）：JR 的 rd 字段=0 → $0 → 写保护拦截

从寄存器文件设计者角度：`$0` 保护是必须的（MIPS 规范要求）。问题不在寄存器文件，而在上游的控制信号——目标寄存器选择逻辑和控制信号生成逻辑。寄存器文件正确完成了"保护 $0 不被写入"的任务，但它无法区分"合法的 $0 寻址"和"错误的 $0 寻址"。这个责任应该在 idecode32 的 write_register_address 生成逻辑中。

### 3. 存在的问题以及解决办法
报告撰写过程中没有遇到新技术问题。主要挑战是如何把技术细节（7 位 ALU_ctl 编码、6 种移位类型的 Verilog 实现差异、两层 ALUOp 映射的硬件设计理由）用清晰的语言表述，让读者（老师）能理解设计意图而不仅仅是代码罗列。

（2）**答辩准备**

梳理我作为角色 B 需要准备的答辩内容：

**核心问题 1：ALU 控制编码是如何生成的？**
回答要点：两层映射——第一层 control32 根据 opcode 生成 ALUOp[1:0]（00=无运算, 01=分支减法, 10=R型由Function决定, 11=I型ALU）；第二层 executs32 根据 ALUOp+Function_opcode 生成最终的 3 位 ALU_ctl。

**核心问题 2：SRA 和 SRL 的区别？Verilog 中如何实现？**
回答要点：SRL 逻辑右移高位填 0，用 `Binput >> Shamt`；SRA 算术右移高位填符号位，用 `$signed(Binput) >>> Shamt`。关键在 `$signed()`——不加的话 `>>>` 和 `>>` 行为相同，算术右移变成逻辑右移。

**核心问题 3：$0 寄存器怎么实现"永远为 0"？**
回答要点：读操作 `register[0]` 不参与任何写操作（`write_register_address != 5'b00000` 条件排除），初始化时 reset=1 所有寄存器清零。**不是靠组合逻辑 `addr==0 ? 0 : register[addr]`**——这样做综合出额外的 MUX，浪费资源。靠的是"永远不写"来保证值一直为 0。

**核心问题 4：为什么移位指令有固定移位和可变移位两种？**
回答要点：固定移位（SLL/SRL/SRA）的移位量编码在指令中的 shamt 字段（instruction[10:6]），可变移位（SLLV/SRLV/SRAV）的移位量来自 rs 寄存器的低 5 位。这给了编译器选择——编译时常量用固定移位（更快），运行时的变量移位用可变移位。

（3）**个人技术收获总结**

1. **ALU 控制编码的两层映射设计**：第一层（opcode→ALUOp）减少了控制信号位宽，第二层（ALUOp+Function→ALU_ctl）在 executs32 内部完成最终译码。这种分层设计平衡了控制器的复杂度和 ALU 的灵活性——如果所有指令都直接从 opcode 映射到 ALU_ctl，control32 的译码逻辑会极其臃肿。

2. **Verilog 有符号数处理**：SRA 的 `$signed(Binput) >>> Shamt` 是我遇到的最有教育意义的 Verilog 问题。`>>>` 和 `>>` 的行为差异取决于操作数的 signed 属性——这不是综合工具的问题，而是 Verilog 语言规范的定义。如果不主动用 `$signed()` 声明，默认无符号数，算术右移等价于逻辑右移——功能完全错误但仿真不会报 warning。

3. **寄存器文件的双读单写架构**：双读端口用组合逻辑（直接索引，无延迟）——指令需要同时读取 rs 和 rt 的值。单写端口用时序逻辑（posedge clock）——每条指令最多写一个寄存器，且写入发生在时钟边沿，便于时序收敛。这是 RISC 架构寄存器文件的标准设计模式。

4. **$0 写保护的双刃剑**：`$0` 保护是 MIPS 架构的正确设计，但它恰好掩盖了 Bug #7 和 Bug #8。从调试角度，如果当时 $0 没有被写保护，两个 Bug 会立即表现为 `$0` 的值异常（不再是 0），反而更容易被发现。这让我认识到：冗余的保护机制在掩盖 Bug 的同时，也延误了 Bug 的发现。正确的做法不是去掉保护，而是在仿真中增加 assertion——比如 `assert(register[0] == 0)`——让试图写 $0 的行为在仿真中报错而非静默丢弃。

5. **LUI 实现的硬件美学**：`{Binput[15:0], 16'b0}` 在硬件中只是一个布线操作——16 根线直连、16 根线接地——不消耗任何逻辑门。这种"零成本"操作在软件中是不可想象的（软件中移位/拼接都需要 CPU 周期），但在硬件中只需要物理连线。这是硬件设计的独特魅力：布线 = 计算。

### 3. 存在的问题以及解决办法
（1）**ANDI/ORI/XORI 零扩展问题**：如前所述，逻辑立即数指令按 MIPS 规范应使用零扩展，当前使用符号扩展在立即数 bit15=1 时可能产生不正确结果。这是一个理论上的潜在问题，当前测试程序未触发。后续如有需要，可在 executs32.v 中增加一个零扩展通路——工作量很小，只需增加一个 `assign zero_extend = {16'b0, imm}` 并在 ALUOp 指示逻辑指令时选择零扩展作为 B 输入。
