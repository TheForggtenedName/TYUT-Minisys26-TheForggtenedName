[上午 10:30] 撰写详细设计 — 存储器系统章节：

  dmemory32.v 详解：
  （1）时钟反相原理：
    wire ram_clk = !ram_clk_i;
    Block RAM 读操作有硬件延迟——地址在时钟上升沿锁存，数据下一个上升沿
    才稳定输出。如果 CPU 在同一个上升沿给出地址又锁存数据，会锁存到上一次的
    值。时钟反相后给了地址半个周期的"提前量"，确保 CPU 锁存到正确数据。

  （2）CPU/UART 双模式切换（kickOff 信号）：
    wire kickOff = upg_rst_i | (~upg_rst_i & upg_done_i);
    kickOff=0（UART 烧写中）：RAM 使用 UART 时钟和地址
    kickOff=1（烧写完成或复位）：RAM 切回 CPU 时钟和地址

  memorio.v 详解——四个核心功能：
  （1）地址透传：assign address = caddress;
       ALU 算出的 32 位地址直接给 RAM，即使访问 IO 也会传过去
       （但 RAM 写使能没开就不会写入，无副作用）
  （2）写数据路由：memwrite=1 或 iowrite=1 → 输出 wdata；否则 → 32'hZZZZZZZZ
       （高阻态——不驱动总线，避免多模块争抢总线）
  （3）读数据路由：memread=1 → 选 RAM 的 mread_data；否则 → IO 的 ioread_data
  （4）IO 片选译码：
       LEDCtrl:    caddress[31:4]==28'hFFFFFC6
       SwitchCtrl: caddress[31:4]==28'hFFFFFC7
       DigitubeCtrl: caddress[31:4]==28'hFFFFFC8（实验二新增）
       取 [31:4] 而非 [31:0] 的原因：低 4 位是外设内部偏移量，
       LED 分两次写（偏移0低16位、偏移4高8位），不应参与基地址比较

  ioread.v 详解：
  - 多路选择器，将各 IO 设备的读数据汇总返回 CPU
  - 实验二新增 digitube 读通道
  - 实验二修复 latch：所有 else 分支赋 16'b0（确定值），而非自赋值

[下午 14:00] 撰写详细设计 — 数据通路整合章节：

  以 ORI $1, $0, 0x1D（机器码 0x3401001D）为例走完整数据通路：

    STEP 1: ifetc32  — PC=0x00 → ROM[0x00] → instruction=0x3401001D
    STEP 2: idecode32 — 拆出 rs=0, rt=1, imm=0x001D
                       读 $0=0, 符号扩展 → sign_extend=0x0000001D
    STEP 3: control32 — opcode=001101 → RegWrite=1, ALUSrc=1, RegDst=0 ...
    STEP 4: executs32 — ALU 算 OR(0, 0x001D) → alu_result=0x0000001D
    STEP 5: memorio   — 地址=0x0000001D, 高22位=0 → RAM 访问, 非 IO
                       LEDCtrl=0, SwitchCtrl=0, DigitubeCtrl=0
    STEP 6: idecode32 — RegWrite=1 → $1 = ALU_result

  "交通枢纽"的角色：访存阶段决定数据往哪里写、从哪里读。
