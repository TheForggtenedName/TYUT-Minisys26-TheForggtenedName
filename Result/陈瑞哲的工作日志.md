（2）理解IO地址译码的核心逻辑：

```verilog
wire is_io = (Alu_resultHigh[21:0] == 22'h3FFFFF);
MemRead  = Lw && !is_io;
IORead   = Lw &&  is_io;
MemWrite = Sw && !is_io;
IOWrite  = Sw &&  is_io;
```

`22'h3FFFFF` 即 22 位全 1，对应地址 `0xFFFFFC00` 及以上。这是 Memory-Mapped I/O 的硬件判断条件——当 ALU 结果地址的高 22 位全为 1 时，CPU 访问的是 IO 空间而非内存空间。

（3）回顾我负责的 ifetc32.v 设计要点：

- **PC+4 计算**：`PC_plus_4[31:2] = PC[31:2] + 1'b1; PC_plus_4[1:0] = 2'b00;`——32 位指令 4 字节对齐，PC+4 = 高 30 位加 1

（2）**idecode32.v Bug #7 修复——JAL 目标寄存器地址**

原代码问题分析（我负责的译码逻辑）：
```verilog
// 实验一代码（有Bug）：
always @* begin
    if (RegDst == 1'b1)
        write_register_address = rd;    // instruction[15:11]
    else
        write_register_address = rt;    // instruction[20:16]
end
```

JAL 指令（opcode=000011）不是 R 型 → RegDst=0 → 走 else 分支 → 选中 `instruction[20:16]`。

但 JAL 指令编码为 `|000011|instr_index[25:0]|`，bit[20:16] 不是 rt 寄存器号——它是 26 位跳转目标地址的一部分。MIPS 规范规定 JAL 隐式写入 `$31`（`$ra`，返回地址寄存器）。

修复后代码：
```verilog
always @* begin
    if (Jal == 1'b1)
        write_register_address = 5'b11111;       // JAL 显式写 $31 ($ra)
    else if (RegDst == 1'b1)
        write_register_address = rd;              // R 型选 rd
    else
        write_register_address = rt;              // I 型选 rt
end
```

这是实验一最危险的潜伏 Bug——`$0` 写保护恰好掩盖了它。修复后 JAL/JR 的完整调用-返回链路正确：JAL 把 PC+4 存入 `$31` → JR 从 `$31` 读出跳转。

（2）**next_PC 优先级链的深入分析**

ifetc32.v 中 next_PC 的优先级设计：

```verilog
if (Jrn == 1'b1)
    next_PC = Read_data_1;                                    // 优先级 1: JR
else if (Branch == 1'b1 && Zero == 1'b1)
    next_PC = Add_result;                                     // 优先级 2: BEQ成立
else if (nBranch == 1'b1 && Zero == 1'b0)
    next_PC = Add_result;                                     // 优先级 3: BNE成立
else if (Jmp == 1'b1 || Jal == 1'b1)
    next_PC = {PC_plus_4[31:28], Instruction[25:0], 2'b00};  // 优先级 4: J/JAL
else
    next_PC = PC_plus_4;                                      // 优先级 5: 顺序执行
```

**为什么是这个优先级顺序？**

- JR 优先级最高：JR 用于函数返回，必须准确跳转到 `$ra` 中的返回地址。如果被其他条件覆盖，程序流会完全混乱。
- BEQ/BNE 次之：分支指令是条件跳转，条件成立时必须跳转。注意 BEQ 和 BNE 的条件互斥（一个要 Zero=1，一个要 Zero=0），同一时刻最多只有一个成立。
- J/JAL 随后：无条件跳转，但优先级低于分支——因为 J/JAL 上一条指令可能是分支延迟槽（虽然这是单周期 CPU 无延迟槽，但优先级设计遵循 MIPS 惯例）。
- 顺序执行兜底：所有跳转条件都不满足时，PC 正常 +4。

**PC 寄存器为什么用下降沿？**

```verilog
always @(negedge clock) begin
    if (reset) begin PC <= 0; opcplus4 <= 0; end
    else      begin PC <= next_PC; opcplus4 <= PC_plus_4; end
end
```

因为寄存器文件的写操作在上升沿（`posedge clock`）——如果同一条指令的 PC 更新也在上升沿，可能在同一个时钟边沿发生"PC 更新 → 取新指令 → 寄存器文件写旧指令结果"的竞态。下降沿更新 PC 将两者错开半个周期，保证寄存器先写完、PC 再跳转。

（3）**J/JAL 跳转地址计算的边界理解**

`{PC_plus_4[31:28], Instruction[25:0], 2'b00}`

- `PC_plus_4[31:28]`：当前指令的下一句地址的高 4 位。跳转目标必须和当前程序在同一 256MB 段内
- `Instruction[25:0]`：J 型指令的 26 位跳转目标（以字为单位，所以能寻址 2^26 × 4 = 256MB）
- `2'b00`：字对齐，每条指令 4 字节

这个设计意味着 J/JAL 只能跳转到与当前 PC+4 处于同一 256MB 区域的地址。对于 Minisys-1 CPU（64KB 程序空间），这个限制完全不是问题。

（4）**13 → 17 控制信号的设计哲学**

为什么实验二需要把 13 个信号扩展到 17 个？核心原因是 **Memory-Mapped I/O 的引入改变了访存信号的语义**：

- 实验一：`MemtoReg=Lw`、`MemWrite=Sw`——两个信号足够，因为只有"内存"一种外部数据源
- 实验二：IO 空间与内存空间共享相同的 LW/SW 指令——必须用地址高位区分访问目标。拆分为 `MemRead/MemWrite`（内存）和 `IORead/IOWrite`（IO），让下游模块精确知道数据往哪里去

这体现了硬件设计的一个重要原则：**控制信号应该直接反映物理操作**，而不是通过其他信号间接推断。

### 3. 存在的问题以及解决办法
（1）**ifetc32.v 中 J/JAL 的 next_PC 逻辑合并**：
```verilog
else if (Jmp == 1'b1 || Jal == 1'b1)
    next_PC = {PC_plus_4[31:28], Instruction[25:0], 2'b00};
```
J 和 JAL 共用同一套地址计算公式——因为两者的跳转目标编码完全相同。区别仅在于 JAL 额外将 `opcplus4`（返回地址）保存到 `$31`，这一操作发生在 idecode32 的写回阶段而非 ifetc32 的取指阶段。这种"就近处理"的设计让每个模块只关心自己需要的逻辑。

### 2. 工作内容
（1）**取指单元（ifetc32.v）章节撰写**

**PC+4 计算原理**：
32 位指令按 4 字节对齐存储，最低 2 位地址恒为 0。因此 PC+4 等效于高 30 位加 1，低 2 位恒为 0。在硬件上，这是一个 30 位加法器 + 2 位接地——不需要 32 位全加器。

**next_PC 五级优先级选择器**：
各级的使能条件、选择的数据源、对应的指令类型，以及优先级排序的理由。

**PC 寄存器设计**：下降沿更新，与寄存器文件的上升沿写操作错开——这是单周期 CPU 避免同一周期内读写冲突的标准方法。

**控制器设计的两个关键决策点**：

决策 1：为什么用 `is_io` 信号而不是直接在每条信号中判断？
```verilog
// 方案A（当前采用）——统一判断：
wire is_io = (Alu_resultHigh[21:0] == 22'h3FFFFF);
MemRead = Lw && !is_io;
IORead  = Lw &&  is_io;

// 方案B（备选）——分散判断：
MemRead = Lw && (Alu_resultHigh[21:0] != 22'h3FFFFF);
IORead  = Lw && (Alu_resultHigh[21:0] == 22'h3FFFFF);
```

选择方案 A 的理由：`is_io` 是一个有物理意义的信号——"当前访问的是 IO 空间"。集中定义后，所有访存信号都引用它，既减少代码重复，也让信号语义更清晰。如果 IO 地址范围发生变化，只需改一处。

决策 2：为什么保持 Jrn 为独立信号而不是合并到 RegWrite？
Jrn 信号有三个用途——控制 next_PC 选择（取指单元）、排除 RegWrite（控制器）、不参与 ALU 运算。如果合并到 RegWrite 内部（即 RegWrite 直接排除 Jrn），那么取指单元需要从 RegWrite=0 + R_format=1 来推断 JR，不如直接用 Jrn 信号明确。

（2）**答辩准备**

梳理我作为角色 A 需要准备的答辩内容：

**核心问题 1：next_PC 的五级优先级是怎么设计的？为什么是这个顺序？**
回答要点：JR > BEQ(Zero=1) > BNE(Zero=0) > J/JAL > PC+4。JR 是函数返回必须精确跳转；BEQ/BNE 条件成立时必须跳转（且两者互斥）；J/JAL 是无条件但优先级低于条件分支（MIPS 惯例）；PC+4 兜底。

**核心问题 2：JAL 和 J 指令的 next_PC 计算一模一样，那它们的区别在哪里？**
回答要点：JAL 和 J 的跳转地址计算完全相同（`{PC_plus_4[31:28], Instruction[25:0], 2'b00}`），区别不在取指单元而在写回阶段——JAL 额外将 `opcplus4`（PC+4）存入 `$31`。ifetc32 的输出 `opcplus4` 就是为 JAL 准备的。这体现了硬件设计中的"职责分离"。

**核心问题 3：17 个控制信号中，哪几个是实验二新增的？为什么要新增？**
回答要点：新增 4 个输出（MemRead/IORead/IOWrite/MemorIOtoReg）和 1 个输入（Alu_resultHigh[21:0]）。原因是 Memory-Mapped I/O——CPU 用同一套 LW/SW 指令访问内存和外设，必须用地址高位区分，控制信号自然要拆分为"内存版本"和"IO 版本"。

**核心问题 4：Bug #7 和 Bug #8 是如何发现的？仿真为什么没暴露？**
回答要点：两个 Bug 都是人工代码审查发现的。仿真没暴露是因为 `$0` 写保护——JAL 的 bit[20:16] 恰好为 0，JR 的 rd 字段恰好为 0，写入目标都是 `$0`，被 `write_register_address != 5'b00000` 条件拦截。换一个测试程序（JAL 跳转地址的 bit[20:16] ≠ 0），Bug #7 立即暴露——随机寄存器被静默写入错误值。

1. **控制器的本质是"查表+译码"**：17 个控制信号每个都是一条布尔表达式，整个控制器就是一张从 {opcode, function_code} 到 {control_signals} 的真值表映射。用组合逻辑实现这张表就是硬件控制器。

2. **指令编码的"隐式信息"是 Bug 的来源**：JAL 隐式写 `$31`、JR 隐式不写寄存器——这些"隐含规则"不体现在指令编码中，而是 MIPS 架构的规范规定。编码阶段必须逐条核对规范，不能仅凭格式分类（"R 型都写寄存器"）来推断。

3. **next_PC 优先级链是程序流的硬件保障**：五级 if-else 看起来是简单的组合逻辑，但每级的选择决定了 CPU 能否正确执行分支、跳转和函数调用。JR 最高优先级的理由是函数返回必须精确——这个设计决策不是随意的，而是由指令语义决定的。

4. **控制器是数据通路和指令集之间的"翻译层"**：控制器的 17 个信号就像 17 根操纵杆——每根操纵杆控制数据通路中的一个多路选择器或使能门。不同的指令组合就是不同的操纵杆位置。控制器的设计质量直接决定了 CPU 能否"正确理解"每一条指令。

5. **好的模块化设计让扩展局部化**：实验二新增了 IO 地址译码和数码管，但 ifetc32.v 一行未改，idecode32.v 只改了 3 行（Bug #7 修复）。因为取指单元不关心"数据往哪去"，译码单元不关心"地址属于内存还是 IO"。职责边界清晰，改动自然隔离。

6. **"仿真通过 ≠ 代码正确"从我负责的模块角度**：Bug #7 和 Bug #8 的根因都在我负责的模块中——目标寄存器选择逻辑和 RegWrite 信号生成逻辑。两个 Bug 分别在 idecode32.v 和 control32.v 中，但都因为 `$0` 写保护而"恰好不暴露"。如果 FPGA 设计有类似软件的 assertion 机制（比如 `assert(rd != 0 || !RegWrite)`），这类问题可以在仿真中第一时间捕获。

### 3. 存在的问题以及解决办法
无遗留问题。我负责的 ifetc32.v、idecode32.v（译码部分）和 control32.v 均已通过全流程验证。两个实验一潜伏 Bug 在实验二中全部修复。唯一的"缺口"是控制器的 17 端口版本需要更多的测试激励来覆盖 IO 地址译码的所有 corner case（比如地址恰好处于 IO 空间边界、越界访问等），当前测试程序只覆盖了正常路径。这属于测试完备性问题，不影响核心功能的正确性。
